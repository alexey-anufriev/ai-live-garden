# AI Live Garden Agent Rules

This repository is an autonomous Java/Maven garden simulation. Each agent run observes the repository, chooses one focused bounded improvement, changes the project, and leaves a committable result. CI scripts record the run afterward.

The goal is not to finish a roadmap. The goal is to see whether autonomous agents can keep a living software system coherent, expressive, and extensible over time.

## Sources Of Truth

- `data/garden-state.txt` is the committed living state. Normally advance it through the program. During a demonstrated runaway, deadlock, corrupt state, or CI-operability emergency, an agent has god-mode recovery authority to modify persisted state deliberately, preferably through a deterministic program command such as `recover`, rather than waiting for ordinary ticks.
- AI-less tick workflows may advance `data/garden-state.txt` and refresh deterministic README/chart state. Treat those commits as real ecological history.
- `agent/state.md`, `agent/code-map.md`, README state, `agent/garden-trends.svg`, `agent/organism-trends.svg`, `agent/journal/`, and `agent/summaries/` are generated by post-processing during autonomous runs. `agent/shadow-feedback.md`, when present, is harness-generated evidence from a previous incomplete or rejected attempt. For a shadow rejection it points to the exact preserved candidate commit on an `agent-rejected/<run>-<attempt>` branch; inspect that commit before deciding what to reuse, revise, or abandon.
- `scripts/find-active-agent-plan.sh` selects the newest dated Project Manager plan only on its own UTC date by default. When an active plan exists and baseline repair is not required, it is the highest product priority. Older plans remain continuity evidence, not binding direction. The age window can be changed explicitly with `AGENT_PM_PLAN_MAX_AGE_DAYS`.
- Archives under `agent/journal/archive/` and `agent/summaries/**/archive/` are retained history; normal runs must not read them.
- `story/` is maintained by the Story workflow. Normal agent runs must not edit it.

## Required Run Flow

Use the compact context from `scripts/build-agent-context.sh` when the workflow provides it. Inspect raw files only when exact details are needed.

Every autonomous run must:

1. Understand `AGENTS.md`, `GEMINI.md`, README, active memory summaries, `data/garden-state.txt`, and the Java/Maven project from the compact context.
2. Check the baseline Maven test result in the compact context. If it failed, make repairing the existing Java source or tests the run's first task.
3. If baseline tests or worktree policy are failing, make repair the focused task and set `pmDirection` to `none`. Otherwise, if `scripts/find-active-agent-plan.sh` returns a plan, choose exactly one listed direction and set `pmDirection` to its label. When it returns no plan, choose from current-state evidence and set `pmDirection` to `none`.
4. Make a meaningful source, test, rendering, or project-file change with a visible expected effect. A handoff-only or generated-memory-only diff is a failed no-op and must not reach the garden tick.
5. Run `scripts/run-maven-tests-with-timeout.sh` if possible. A timeout or interrupt is a failing baseline, not permission to claim success or continue unrelated work.
6. Write `.agent-run.json` as the machine-readable handoff.
7. Do not edit generated memory files.
8. Leave no scratch files, unrelated edits, or misleading test claims.

Autonomous workflow runs are pre-approved execution runs. Do not enter planning mode, ask for confirmation, or stop after proposing a strategy; implement the focused task or fail with a clear handoff.

After the agent step, the workflow advances `data/garden-state.txt` and runs `scripts/agent-auto-postprocess.sh`. That script parses `.agent-run.json`, restores generated memory files, updates README state and its embedded trend charts, rewrites `agent/state.md`, refreshes `agent/code-map.md` from the Java tree, appends summaries, creates the journal entry, appends any request entries, removes `.agent-run.json`, and then validators check the result.

## Choosing Work

Prefer changes that add durable garden value:

- clearer ecological behavior;
- stronger feedback loops;
- recoverable missing roles;
- state transitions that respond to the current garden;
- behavior-supporting boundaries for future work;
- observable consequences in future ticks.

Unless the run is repairing a failing baseline, the task must change future simulation behavior, rendered garden state, or persisted state semantics in a way that can be observed in future ticks. Do not choose behavior-neutral refactoring, centralization, extraction, renaming, relocation, or cleanup as the main task. Those edits are allowed only as supporting changes needed for a behavior-changing task or as part of required repair.

Architecture-budget exception: when `scripts/report-complexity-budget.sh` reports an exceeded advisory threshold, no baseline repair or active PM plan exists, and the task measurably reduces that exceeded metric, one bounded behavior-preserving consolidation may be the focused task. It must use a `preserve` shadow target, retain the same seeded ecological outcome within its declared tolerance, and avoid unrelated cleanup. This exception exists to prevent endless trait/test accretion from destroying future autonomous capacity.

A run that only adds a named trait, diagnostic field, renderer line, event-log message, counter, or test coverage is low value unless it changes future simulation behavior or removes a concrete obstacle. If a candidate change would only add a name, counter, log line, renderer phrase, or isolated test, choose a stronger task.

`expectedGardenEffect` must describe a future ecological consequence, not maintainability. Invalid effects include "no immediate behavioral change", "easier maintenance", "cleaner code", "centralized logic", "better extensibility", and "simpler future changes".

Focused does not mean tiny. A bounded medium improvement may span several files when it has one clear behavioral purpose and leaves the project coherent. Create a focused new source, test, renderer, fixture, report, or helper when that is clearer than overloading an existing file.

When changing behavior, add or update a focused test unless the change is purely documentation or too narrow to test meaningfully. Tests must prove behavior, not just wording. A tests-only run is valid only when it exposes an important current uncertainty and the handoff explains why no behavior change was appropriate. Do not weaken unrelated assertions.

Tests should have behavior-specific names and should exercise simulation behavior. Do not add tests that only verify generic value storage, trait-list membership, event wording, or a long loop that waits for a lucky condition. Remove scratch reasoning comments before finishing.

Good bounded medium tasks include making an overfull nutrient buffer change release behavior under drought or starvation, making a missing ecological role recoverable from the current state, making fungi, moss, roots, predators, or herbivores respond to measurable environmental stress, making reproduction, starvation, predation, or decay depend on existing garden state in a new observable way, or making state-format evolution explicit when it changes persisted simulation meaning.

## Emergency Recovery Authority

Agents may act god-like when the living state itself prevents the system from testing, ticking, or recovering autonomously. This authority includes bounded deterministic culling, rebalancing, reseeding, state migration, removal of corrupt organisms, and direct persisted-state repair.

Use this authority only when evidence shows an operability emergency. Preserve every viable ecological role when possible, keep the cycle and identifiers coherent, record before/after counts in `.agent-run.json` evidence, and add a garden event explaining the intervention. Prefer `mvn -B -q exec:java -Dexec.args="recover --max-organisms 12000"` over hand-editing thousands of organism lines.

Do not preserve a catastrophic state merely because it is historical. Do not erase or fabricate history casually either: make the smallest deterministic intervention that restores bounded execution, and pair it with a source-level guard that prevents recurrence.

Tests and simulation commands must remain bounded. Do not add long loops that wait for random events or advance a reproducing garden for hundreds of cycles. Use injected randomness, direct phase calculators, and small fixtures. If a test, Maven process, or simulation exceeds its timeout, allow the harness to interrupt it, diagnose the active phase, and repair the cause before continuing.

## Scope Rules

Change only files needed for the chosen task.

Do not edit:

- generated memory files: `README.md`, `agent/state.md`, `agent/requests.md`, `agent/code-map.md`, `agent/shadow-feedback.md`, `agent/garden-trends.svg`, `agent/organism-trends.svg`, `agent/journal/`, `agent/summaries/`, or `agent/templates/`;
- Project Manager direction files under `agent/plans/`;
- old journal entries or archives;
- unrelated tests or behavior;
- `story/`;
- license files;
- secrets or secret-related configuration.

Autonomous agents must not modify `AGENTS.md`, `GEMINI.md`, or `.github/` unless the human explicitly changes the experiment boundary.

If a task needs a dependency, external service, CI change, protected-file change, permission change, or clarification, choose a smaller allowed task. Emergency persisted-state recovery under the authority above is not considered a forbidden generated-memory edit.

Final worktree validation rejects generated bytecode, binary artifacts, root scratch files, generic new test names, and tests with scratch reasoning or trivial trait-storage assertions. Keep compiled output under `target/` only.

## Automated Memory

Autonomous runs should not hand-format README state, state memory, summaries, or journal entries.

Before finishing, agents must write `.agent-run.json`:

```json
{
  "title": "Short title for this run",
  "task": "One sentence describing the chosen implementation task.",
  "why": "Why this was the right next step.",
  "summary": "What changed.",
  "observations": "What was learned or any limitations.",
  "next": "One possible next direction.",
  "expectedGardenEffect": "What future ticks should do differently because of this change.",
  "pmDirection": "A, B, C, D, or none when no PM plan exists or this run is required repair.",
  "evidence": {
    "bottleneck": "The concrete current-state gate or missing causal link addressed.",
    "currentState": "Evidence that the change can affect the committed living state.",
    "verification": "The focused test or bounded command and its observed result."
  },
  "evaluation": {
    "metric": "population.BEETLE, another population type, totalOrganisms, nutrients, nutrientBuffer, or tests for repair only",
    "goal": "increase, decrease, preserve, or pass for repair only",
    "requiredDelta": 1
  },
  "codeMap": [
    {
      "path": "src/main/java/garden/ai/FileChangedByThisRun.java",
      "description": "Short current responsibility of this source file."
    }
  ],
  "requests": [],
  "state": {
    "immediateDirections": [],
    "constraints": []
  }
}
```

`codeMap` should include new or meaningfully changed Java source files from the run, with one short stable responsibility description each. Describe what the file is for, not what changed in the current run. Avoid change-specific phrasing such as "updated", "added", "implemented", "now supports", or "new logic".

`evaluation` declares the deterministic shadow-simulation target. Normal runs must use an ecological metric and a positive delta for increase/decrease goals. Required baseline repairs use `tests`, `pass`, and `0`. CI compares baseline and candidate code from the same state and seeds before advancing or committing the garden.

`requests` may contain objects with `title`, `what`, `why`, `affected`, `benefit`, and `fallback`. Use an empty array when no human help is needed.

The deterministic memory scripts are:

- `scripts/agent-auto-postprocess.sh`
- `scripts/generate-garden-trends-svg.sh`
- `scripts/update-readme-garden-state.sh`
- `scripts/agent-update-code-map.sh`
- `scripts/agent-update-readme-state.sh`
- `scripts/agent-append-summary.sh`
- `scripts/agent-create-journal-entry.sh`

README health uses the protected block between:

```markdown
<!-- AI-LIVE-GARDEN:STATE-START -->
<!-- AI-LIVE-GARDEN:STATE-END -->
```

Allowed health statuses:

- 🟢 Flourishing: diverse, growing, reproducing, no major collapse visible.
- 🟡 Stable: alive and coherent, but limited growth or minor pressure.
- 🟠 Strained: stress, hunger, pressure, or declining diversity.
- 🔴 Critical: collapse risk, starvation, extinction, or broken food chain.
- ⚫ Dormant: insufficient data, inactive garden, or unclear state.

Generated summaries are append-only. If earlier content is wrong or malformed, post-processing appends new information instead of editing history.

Only the latest 100 journal entries should remain directly under `agent/journal/`; older entries may be moved unchanged into `agent/journal/archive/`.

The workflow archives old summaries unchanged: 100 daily, 50 weekly, 12 monthly, and 10 yearly files remain active.

## Workflow And Test Recovery

The Agent workflow calls Gemini through `GEMINI_API_KEY` and passes `EXECUTION_MODEL`, currently `gemini-3.1-flash-lite`.

If Gemini fails because of quota, billing, authentication, or provider availability, the run must fail without committing. Do not work around capacity failures by committing partial changes, suppressing errors, or adding unbounded retries.

The workflow captures bounded Maven and shadow-simulation baselines before the agent step and includes their results in the prompt. If committed Java tests or bounded shadow simulation fail, the agent must repair that baseline first with `pmDirection` set to `none`. The harness does not make same-run corrective agent calls. An incomplete agent result is discarded and recorded as feedback for the next run. Provider quota, authentication, billing, and model-availability failures still fail without committing. After the agent call, CI runs one bounded test validation. If post-change compilation or tests fail, the workflow records that outcome, skips the garden tick, commits the failed source state and deterministic memory, and leaves the failed baseline for the next autonomous run to repair.

Normal evolution also requires the observation window configured by `AGENT_MIN_TICKS_BETWEEN_RUNS` (three AI-less ticks by default). A still-active window is an expected successful no-op, while required repair bypasses the cooldown. Passing candidate code is shadow-simulated from the same state and seeds as the baseline. A target, safety, or operability miss preserves the exact candidate on a run-specific `agent-rejected/` branch, removes it from main, skips the real garden tick, records the branch, commit, handoff, and compact deterministic result in `agent/shadow-feedback.md`, and commits that feedback without failing the workflow. The next autonomous run receives this evidence in its compact context and may inspect or selectively reuse the preserved commit. A later rejection supersedes obsolete rejection branches; a shadow-verified run removes the consumed feedback and rejection branches.

Provider failures, a missing or invalid final `.agent-run.json`, absence of a substantive agent change, deterministic memory validation failures, protected-file violations, and hard final worktree policy violations fail the run without committing. Only post-change Maven compilation/test failure or explicitly deferred worktree policy failure is committed for the next autonomous repair run.

Journal, summary, state, and README formatting are produced by deterministic scripts and validated directly.

## Story Workflow

The Story workflow appends narrative chapters under `story/` from committed garden-state changes. Gemini writes `.story-run.json`; `scripts/story-append-chapter.sh` creates or appends the volume chapter, refreshes `story/chronicles-of-ai-garden.md`, updates `story/last-narrated-garden-state.txt`, and removes the handoff file.

Volume files contain up to 100 chapters. Chapter headings must start with `## Chapter N`.

## Principle

Leave the garden slightly more alive, coherent, inspectable, or maintainable than before.
