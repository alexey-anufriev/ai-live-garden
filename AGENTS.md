# AI Live Garden Agent Rules

This repository is an autonomous Java/Maven garden simulation. Each agent run observes the repository, chooses one focused bounded improvement, changes the project, and leaves a committable result. CI scripts record the run afterward.

The goal is not to finish a roadmap. The goal is to see whether autonomous agents can keep a living software system coherent, expressive, and extensible over time.

## Sources Of Truth

- `data/garden-state.txt` is the committed living state. Normally advance it through the program. During a demonstrated runaway, deadlock, corrupt state, or CI-operability emergency, an agent has god-mode recovery authority to modify persisted state deliberately, preferably through a deterministic program command such as `recover`, rather than waiting for ordinary ticks.
- AI-less tick workflows may advance `data/garden-state.txt` and refresh deterministic README/chart state. Treat those commits as real ecological history.
- `agent/state.md`, `agent/code-map.md`, README state, `agent/garden-trends.svg`, `agent/organism-trends.svg`, `agent/journal/`, and `agent/summaries/` are generated by post-processing during autonomous runs. `agent/shadow-feedback.md`, when present, is harness-generated evidence from the previous measured experiment, incomplete attempt, or rejected candidate. Substantive incomplete candidates and hard-gate rejections point to exact preserved commits on `agent-rejected/<run>-<attempt>` branches; inspect those commits before deciding what to reuse, revise, or abandon.
- `scripts/find-active-agent-plan.sh` selects the newest dated Project Manager plan only on its own UTC date by default. When an active plan exists, it is the highest product priority for `evolution` runs. Required repair/recovery and objectively eligible maintenance or diagnostic modes may bypass it. Older plans remain continuity evidence, not binding direction. The age window can be changed explicitly with `AGENT_PM_PLAN_MAX_AGE_DAYS`.
- Archives under `agent/journal/archive/` and `agent/summaries/**/archive/` are retained history; normal runs must not read them.
- `story/` is maintained by the Story workflow. Normal agent runs must not edit it.

## Required Run Flow

Use the compact context from `scripts/build-agent-context.sh` when the workflow provides it. Inspect raw files only when exact details are needed.

Every autonomous run must:

1. Understand `AGENTS.md`, `GEMINI.md`, README, active memory summaries, `data/garden-state.txt`, and the Java/Maven project from the compact context.
2. Check the baseline Maven test result in the compact context. If it failed, make repairing the existing Java source or tests the run's first task.
3. Classify the work with `runMode`: `repair` for a failing test or worktree baseline, `recovery` for a failed bounded shadow baseline or demonstrated persisted-state emergency, `maintenance` for an eligible architecture-budget consolidation, `diagnostic` for focused test-only uncertainty reduction, or `evolution` for normal behavior change. Non-evolution modes set `pmDirection` to `none`. An evolution run must choose one direction from an active PM plan, when present.
4. Make a meaningful source, test, rendering, or project-file change with a visible expected effect. A handoff-only or generated-memory-only diff is a failed no-op and must not reach the garden tick.
5. Run `scripts/run-maven-tests-with-timeout.sh` if possible. A timeout or interrupt is a failing baseline, not permission to claim success or continue unrelated work.
6. Write `.agent-run.json` as the machine-readable handoff.
7. Do not edit generated memory files.
8. Leave no scratch files, unrelated edits, or misleading test claims.

Autonomous workflow runs are pre-approved execution runs. Do not enter planning mode, ask for confirmation, or stop after proposing a strategy; implement the focused task or fail with a clear handoff.

After the agent step, the workflow normally advances `data/garden-state.txt` and runs `scripts/agent-auto-postprocess.sh`. That script parses `.agent-run.json`, restores generated memory files, updates README state and its embedded trend charts, rewrites `agent/state.md`, refreshes `agent/code-map.md` from the Java tree, appends summaries, creates the journal entry, appends any request entries, removes `.agent-run.json`, and then validators check the result. If that post-acceptance tick or generated-memory transaction fails, CI rolls it back and still commits the already validated source candidate with its measured verdict so the next run can continue from real progress.

## Choosing Work

Prefer changes that add durable garden value:

- clearer ecological behavior;
- stronger feedback loops;
- recoverable missing roles;
- state transitions that respond to the current garden;
- behavior-supporting boundaries for future work;
- observable consequences in future ticks.

An `evolution` task must change future simulation behavior, rendered garden state, or persisted state semantics in a way that can be observed in future ticks. Do not label behavior-neutral refactoring, centralization, extraction, renaming, relocation, or cleanup as evolution. Those edits are allowed as supporting changes, required repair, or eligible maintenance.

Explicit non-evolution modes are narrow exceptions, not loopholes:

- `repair` requires a failing baseline and restores it. Test-only repair skips shadow simulation; production repair receives safety-only shadow validation.
- `recovery` requires a failed bounded shadow baseline or a deliberate persisted-state emergency intervention. It receives safety-only validation and may advance the garden after passing.
- `maintenance` requires an objectively exceeded architecture budget and measurably reduces it with a bounded behavior-preserving change. It receives safety-only validation and does not advance the garden.
- `diagnostic` is limited to focused test-only changes that resolve an important current uncertainty. It skips shadow simulation and does not advance the garden.

Architecture-budget exception: when `scripts/report-complexity-budget.sh` reports an exceeded advisory threshold, no baseline repair is required, and the task measurably reduces that exceeded metric, one bounded behavior-preserving `maintenance` consolidation may bypass an active PM plan. It must use a `preserve` target, pass safety-only shadow validation, and avoid unrelated cleanup. This exception exists to prevent endless trait/test accretion from destroying future autonomous capacity.

A run that only adds a named trait, diagnostic field, renderer line, event-log message, counter, or test coverage is low value unless it changes future simulation behavior or removes a concrete obstacle. If a candidate change would only add a name, counter, log line, renderer phrase, or isolated test, choose a stronger task.

`expectedGardenEffect` must describe a future ecological consequence, not maintainability. Invalid effects include "no immediate behavioral change", "easier maintenance", "cleaner code", "centralized logic", "better extensibility", and "simpler future changes".

Focused does not mean tiny. A bounded medium improvement may span several files when it has one clear behavioral purpose and leaves the project coherent. Create a focused new source, test, renderer, fixture, report, or helper when that is clearer than overloading an existing file.

When changing behavior, add or update a focused test unless the change is purely documentation or too narrow to test meaningfully. Tests must prove behavior, not just wording. A tests-only run is valid only when it exposes an important current uncertainty and the handoff explains why no behavior change was appropriate. Do not weaken unrelated assertions.

Tests should have behavior-specific names and should exercise simulation behavior. Do not add tests that only verify generic value storage, trait-list membership, event wording, or a long loop that waits for a lucky condition. Remove scratch reasoning comments before finishing.

Good bounded medium tasks include making an overfull nutrient buffer change release behavior under drought or starvation, making a missing ecological role recoverable from the current state, making fungi, moss, roots, predators, or herbivores respond to measurable environmental stress, making reproduction, starvation, predation, or decay depend on existing garden state in a new observable way, or making state-format evolution explicit when it changes persisted simulation meaning.

## Emergency Recovery Authority

Agents may act god-like when the living state itself prevents the system from testing, ticking, or recovering autonomously. This authority includes bounded deterministic culling, rebalancing, reseeding, state migration, removal of corrupt organisms, and direct persisted-state repair.

Use this authority only when evidence shows an operability emergency. Preserve every viable ecological role when possible, keep the cycle and identifiers coherent, record before/after counts in `.agent-run.json` evidence, and add a garden event explaining the intervention. Prefer `mvn -B -q exec:java -Dexec.args="recover --max-organisms 12000"` over hand-editing thousands of organism lines.

Do not preserve a catastrophic state merely because it is historical. Do not erase or fabricate history casually either: make the smallest deterministic intervention that restores bounded execution, and pair it with a source-level guard that prevents recurrence.

Tests and simulation commands must remain bounded. Do not add long loops that wait for random events or advance a reproducing garden for hundreds of cycles. Use injected randomness, direct phase calculators, and small fixtures. If a test, Maven process, or simulation exceeds its timeout, allow the harness to interrupt it, diagnose the active phase, and repair the cause before continuing.

## Scope Rules

Change only files needed for the chosen task.

Do not edit:

- generated memory files: `README.md`, `agent/state.md`, `agent/requests.md`, `agent/code-map.md`, `agent/shadow-feedback.md`, `agent/garden-trends.svg`, `agent/organism-trends.svg`, `agent/journal/`, `agent/summaries/`, or `agent/templates/`;
- Project Manager direction files under `agent/plans/`;
- old journal entries or archives;
- unrelated tests or behavior;
- `story/`;
- license files;
- secrets or secret-related configuration.

Autonomous agents must not modify `AGENTS.md`, `GEMINI.md`, or `.github/` unless the human explicitly changes the experiment boundary.

If a task needs a dependency, external service, CI change, protected-file change, permission change, or clarification, choose a smaller allowed task. Emergency persisted-state recovery under the authority above is not considered a forbidden generated-memory edit.

Final worktree validation rejects generated bytecode, binary artifacts, root scratch files, generic new test names, and tests with scratch reasoning or trivial trait-storage assertions. Keep compiled output under `target/` only.

## Automated Memory

Autonomous runs should not hand-format README state, state memory, summaries, or journal entries.

Before finishing, agents must write `.agent-run.json`:

```json
{
  "runMode": "evolution",
  "acceptanceSource": "agent",
  "title": "Short title for this run",
  "task": "One sentence describing the chosen implementation task.",
  "why": "Why this was the right next step.",
  "summary": "What changed.",
  "observations": "What was learned or any limitations.",
  "next": "One possible next direction.",
  "expectedGardenEffect": "What future ticks should do differently because of this change.",
  "pmDirection": "A, B, C, or D for evolution with an active PM plan; otherwise none.",
  "evidence": {
    "bottleneck": "The concrete current-state gate or missing causal link addressed.",
    "currentState": "Evidence that the change can affect the committed living state.",
    "verification": "Focused tests plus the shadow preflight baseline average, candidate average, and observedDelta."
  },
  "evaluation": {
    "metric": "population.BEETLE, another population type, totalOrganisms, nutrients, nutrientBuffer, or tests for repair only",
    "goal": "increase, decrease, preserve, or pass for repair only",
    "requiredDelta": 1
  },
  "causalReach": {
    "mechanism": "The concrete production mechanism changed.",
    "traits": ["exact-trait-name"],
    "carrierBasis": "existing, adoption, or not-applicable",
    "activeCarrierCount": 1,
    "adoptionPath": "Where carriers exist or how organisms acquire the trait.",
    "estimatedPhaseImpact": "Numeric current-state impact versus the dominant competing term.",
    "clampRisk": "none, lower, upper, or unknown",
    "previousFeedbackDecision": "reuse, revise, abandon, or none",
    "preflight": {
      "passed": true,
      "observedDelta": 1
    }
  },
  "codeMap": [
    {
      "path": "src/main/java/garden/ai/FileChangedByThisRun.java",
      "description": "Short current responsibility of this source file."
    }
  ],
  "requests": [],
  "state": {
    "immediateDirections": [],
    "constraints": []
  }
}
```

The handoff expresses experiment intent, while CI owns mechanically derivable metadata. Before validation, the harness may reclassify an ineligible colloquial `recovery` source change as `evolution`, canonicalize mode-specific `pmDirection`, `acceptanceSource`, and evaluation fields, infer an unambiguous PM direction from the declared ecological metric, and recalculate trait-carrier counts from the committed garden state. These normalizations are recorded in `harnessNormalization`. Do not reject safe substantive code solely because the model misstated metadata that the harness can derive deterministically; tests, meaningful worktree policy, executable measurement, and population safety remain hard gates. Deferred feedback retains the latest attempt and one immediately preceding attempt without recursively embedding the entire rejection history.

`codeMap` should include new or meaningfully changed Java source files from the run, with one short stable responsibility description each. Describe what the file is for, not what changed in the current run. Avoid change-specific phrasing such as "updated", "added", "implemented", "now supports", or "new logic".

`evaluation` declares the experiment's measurement target. Evolution runs use an ecological metric and a positive delta for increase/decrease goals. Prefer a task-specific target with `acceptanceSource: agent`; choose the smallest nontrivial delta that would prove the mechanism reaches the metric in the bounded window. A PM plan's structured default is a desired fallback outcome, not a mandatory choice, and must match exactly when `acceptanceSource: pm`. Repair, recovery, and diagnostic modes use `tests`, `pass`, and `0`; maintenance uses an ecological `preserve` target. CI derives target, safety-only, or skipped shadow evaluation from the mode. Evolution measurement is a same-state, same-seed differential: CI averages the final declared metric across candidate seeds and baseline seeds, then calculates `observedDelta = candidateAverage - baselineAverage`. Initial-to-final movement that also occurs in the baseline is not candidate impact.

Evolution runs also require `causalReach`. When the mechanism depends on traits, list their exact names and declare whether the current committed state already has carriers or the same change provides an adoption path. `scripts/count-garden-trait-carriers.sh` is the authoritative carrier count. Global mechanisms use `not-applicable` with no traits. Estimate the mechanism's numeric phase impact against the dominant competing term and record clamp risk. When previous feedback exists, explicitly choose `reuse`, `revise`, or `abandon`. Run the differential shadow preflight when possible and never invent its result. The harness independently measures every candidate and writes the authoritative `observedDelta` into the final handoff; unit tests alone do not measure ecological impact. Safe coherent code is committed even when it misses the declared target, with a `target-met`, `partial-progress`, `inert`, or `wrong-direction` verdict recorded for the next run. Tests, worktree policy, executable measurement, and population safety remain hard gates.

`requests` may contain objects with `title`, `what`, `why`, `affected`, `benefit`, and `fallback`. Use an empty array when no human help is needed.

The deterministic memory scripts are:

- `scripts/agent-auto-postprocess.sh`
- `scripts/generate-garden-trends-svg.sh`
- `scripts/update-readme-garden-state.sh`
- `scripts/agent-update-code-map.sh`
- `scripts/agent-update-readme-state.sh`
- `scripts/agent-append-summary.sh`
- `scripts/agent-create-journal-entry.sh`

README health uses the protected block between:

```markdown
<!-- AI-LIVE-GARDEN:STATE-START -->
<!-- AI-LIVE-GARDEN:STATE-END -->
```

Allowed health statuses:

- 🟢 Flourishing: diverse, growing, reproducing, no major collapse visible.
- 🟡 Stable: alive and coherent, but limited growth or minor pressure.
- 🟠 Strained: stress, hunger, pressure, or declining diversity.
- 🔴 Critical: collapse risk, starvation, extinction, or broken food chain.
- ⚫ Dormant: insufficient data, inactive garden, or unclear state.

Generated summaries are append-only. If earlier content is wrong or malformed, post-processing appends new information instead of editing history.

Only the latest 100 journal entries should remain directly under `agent/journal/`; older entries may be moved unchanged into `agent/journal/archive/`.

The workflow archives old summaries unchanged: 100 daily, 50 weekly, 12 monthly, and 10 yearly files remain active.

## Workflow And Test Recovery

The Agent workflow calls Gemini through `GEMINI_API_KEY` and passes `EXECUTION_MODEL`. Evolution defaults to the known-working `gemini-3.1-flash-lite`; repository variable `AGENT_EXECUTION_MODEL` can opt into a stronger model after its API access, billing, and quota are confirmed without editing the workflow.

If Gemini fails because of quota, billing, authentication, or provider availability, the run must fail without committing. Do not work around capacity failures by committing partial changes, suppressing errors, or adding unbounded retries.

The workflow captures bounded Maven and shadow-simulation baselines before the agent step and includes their results in the prompt. If committed Java tests or bounded shadow simulation fail, the agent must repair that baseline first with `pmDirection` set to `none`. Each run makes exactly one agent call. CI snapshots its substantive candidate, independently runs the executable contract, and classifies measured ecological impact. Safe coherent code is committed after that one call whether the target is met, partially approached, unchanged, or moved in the wrong direction. The verdict becomes mandatory input for the next run, which must decide whether to keep, revise, or revert the implementation already on main. Provider quota, authentication, billing, and model-availability failures are never treated as candidate failures and still fail without committing.

If the single candidate is invalid, fails tests or worktree policy, cannot be measured, or violates population safety, CI preserves substantive source on a rejected-candidate branch, records the structured failure for the next run, removes unvalidated source from main, skips the garden tick, and completes without failing the build. Non-publishable residue is discarded but its bounded diagnostics remain in feedback. The attempt assessor is the single authoritative shadow evaluator; downstream steps repeat tests and worktree policy but do not run a contradictory second shadow gate. The workflow never claims safety or impact from the agent's report alone.

Normal evolution also requires the observation window configured by `AGENT_MIN_TICKS_BETWEEN_RUNS` (three AI-less ticks by default). A still-active window is an expected successful no-op, while required repair bypasses the cooldown. Passing candidate code is shadow-simulated from the same state and seeds as the baseline. Increase and decrease targets are met only at the declared signed differential; preserve targets are met only within the declared absolute tolerance. A safe target miss is not a merge rejection: CI commits the code, advances the garden, and writes the measured verdict to `agent/shadow-feedback.md`. The next autonomous run receives that verdict and must explicitly decide whether to reuse, revise, or abandon the current implementation before selecting work. Inert or wrong-direction code should normally be corrected or reverted before adding another disconnected mechanism. Candidate capture failure, population-safety failure, invalid handoff, test failure, or policy violation still rejects the candidate. Such a rejection preserves exact substantive source on a run-specific `agent-rejected/` branch and records how to inspect it. A later safe experiment supersedes prior verdict feedback and removes obsolete rejection branches.

Provider failures and protected-file violations before candidate acceptance fail the run without committing. Missing or invalid handoffs, absent substantive changes, test failures, unmeasured candidates, and unsafe shadow results produce deterministic feedback rather than a broken main commit. After a candidate has passed its executable contract, a tick, deterministic-memory, archive, journal, summary, or final-worktree failure rolls back that incomplete post-processing transaction and commits only the validated source plus the measured verdict. The next run receives the failure reason and continues; an allowed candidate is not discarded because bookkeeping failed. A measured safe target miss is an experiment result, not a candidate failure.

Journal, summary, state, and README formatting are produced by deterministic scripts and validated directly.

## Story Workflow

The Story workflow appends narrative chapters under `story/` from committed garden-state changes. Gemini writes `.story-run.json`; `scripts/story-append-chapter.sh` creates or appends the volume chapter, refreshes `story/chronicles-of-ai-garden.md`, updates `story/last-narrated-garden-state.txt`, and removes the handoff file.

Volume files contain up to 100 chapters. Chapter headings must start with `## Chapter N`.

## Principle

Leave the garden slightly more alive, coherent, inspectable, or maintainable than before.
